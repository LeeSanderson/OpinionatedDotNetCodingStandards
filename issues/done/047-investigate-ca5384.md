## What to build

Deep-analyse CA5384 ("Do Not Use Digital Signature Algorithm (DSA)") to determine why it is currently untestable, then either fix the test so it passes or update the Untestable note with a confirmed, well-sourced reason.

**Current state:** The test method `ProhibitDigitalSignatureAlgorithmUsage` in the test suite is marked `[Fact(Skip = "untestable")]`. The current Untestable reason is: "Rule does not fire in Microsoft.CodeAnalysis.NetAnalyzers 10.0.x for DSA.Create() nor for DSA.SignData() calls; the abstract factory pattern for DSA appears to not trigger the diagnostic in this analyzer version"

## Current test code

```csharp
    [Fact(Skip = "untestable")]
    [RuleDoc("CA5384", "Do Not Use Digital Signature Algorithm (DSA)",
        HelpLink = "https://learn.microsoft.com/dotnet/fundamentals/code-analysis/quality-rules/ca5384",
        Untestable = "Rule does not fire in Microsoft.CodeAnalysis.NetAnalyzers 10.0.x for DSA.Create() nor for DSA.SignData() calls; the abstract factory pattern for DSA appears to not trigger the diagnostic in this analyzer version")]
    public async Task ProhibitDigitalSignatureAlgorithmUsage()
    {
        using var project = await CreateProjectBuilder();
        await project.AddFile(
            "Program.cs",
            """
            using System.Security.Cryptography;
            namespace test;
            public static class Program
            {
                public static byte[] SignData(byte[] data)
                {
                    using var dsa = DSA.Create();
                    return dsa.SignData(data, HashAlgorithmName.SHA256);
                }
                public static int Main() => 0;
            }
            """);
        var buildOutput = await project.BuildAndGetOutput();

        buildOutput.HasError("CA5384").ShouldBeTrue();
    }
```

3. Test on NetAnalyzers 8.x by temporarily downgrading the `Microsoft.CodeAnalysis.NetAnalyzers` package reference in the test harness and re-running the same `DSA.Create()` pattern to determine whether the diagnostic fired in an earlier version and was later suppressed or removed.

5. Inspect the SARIF / build output for any suppression or severity override: confirm that CA5384 is not being downgraded to a suggestion or hidden by a default editorconfig severity entry generated by `CreateProjectBuilder`, which would prevent `HasError("CA5384")` from matching even if the diagnostic fires at a lower severity.

6. Search the `dotnet/roslyn-analyzers` issue tracker and changelog for CA5384 regressions or intentional scope reductions introduced between versions 8.x and 10.x; a linked GitHub issue would provide a citable source for the Untestable reason.

7. If none of the above yield a firing diagnostic, document the confirmed root cause — including the specific analyzer source file or GitHub issue — and update the `Untestable` property on `[RuleDoc]` with the precise, sourced explanation so future investigators have a clear starting point.
