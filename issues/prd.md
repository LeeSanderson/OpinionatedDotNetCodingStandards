# PRD: Add SonarAnalyzer.CSharp as a fifth bundled analyzer

## Problem Statement

`Opinionated.DotNet.CodingStandards` bundles four Roslyn analyzers (Meziantou,
BannedApiAnalyzers, NetAnalyzers, StyleCop) and enforces a curated set of rules on every
consuming project. Between them these four cover quality, style, security, and banned-API
concerns well — but there is a clear, structural gap they **cannot** fill: **none of them
measure code complexity.** There is no cognitive-complexity, cyclomatic-complexity,
nesting-depth, method-length, or parameter-count enforcement anywhere in the current rule
set, and no amount of configuration of the existing four produces it. Beyond complexity,
Sonar also carries a body of maintainability and design rules that add a useful second
dimension on top of the existing baseline.

As the package maintainer, I want to close that gap by adding Sonar's analyzer, but I need
to do it without (a) drowning consumers in hundreds of new build errors, (b) double-reporting
concerns the existing four already own, or (c) silently introducing a license that downstream
(often commercial) consumers would want to know about.

## Solution

Add `SonarAnalyzer.CSharp` as a fifth bundled analyzer, wired into the existing
analyzer-ingestion pipeline exactly like the other four. Because Sonar ships hundreds of
rules and the package sets `TreatWarningsAsErrors=true`, the analyzer is integrated on an
**opt-in** basis (mirroring the existing StyleCop treatment, where 196 of 197 rules are
disabled): every Sonar rule is listed in a generated per-analyzer editorconfig and seeded to
`none`, and a deliberately chosen **broader maintainability set** (~50–100 rules) is promoted
to enforced severity. Promotion is governed by an **"existing analyzer wins"** overlap policy
— where a Sonar rule is an exact twin of a rule the other four already enforce, the Sonar
rule stays `none`; Sonar is enabled only where it is additive (a new dimension, led by the
complexity metrics) or materially distinct.

The work is captured as a full PRD broken into per-rule issues and delivered as a **single
pull request**, with the high-volume curation-and-test effort produced via the repository's
established **two-phase batch-orchestration workflow** (parallel read-only investigation +
adversarial verification → serial apply/build/test/commit on the main checkout).

## User Stories

1. As a consuming developer, I want cognitive-complexity enforcement (Sonar's flagship rule),
   so that overly convoluted methods are flagged at build time — something no existing
   bundled analyzer provides.
2. As a consuming developer, I want cyclomatic-complexity, nesting-depth, method-length, and
   parameter-count limits enforced, so that structural complexity is caught early.
3. As a consuming developer, I want a curated set of Sonar maintainability and design rules
   enforced on top of the existing baseline, so that I get a second, complementary dimension
   of code-quality feedback.
4. As a consuming developer, I want **no** double-reported diagnostics, so that a single
   concern never produces two build errors with two different messages and help links.
5. As a consuming developer, I want the addition to be safe for my build, so that I am not
   suddenly hit by hundreds of new warnings-as-errors from rules that were never curated.
6. As a consuming developer, I want each newly enforced Sonar rule to behave like the
   existing rules (severity configurable via editorconfig, opt-out where appropriate), so
   that the experience is consistent across all five analyzers.
7. As a downstream/commercial consumer, I want the LGPL-3.0 license of the Sonar analyzer
   disclosed in the README and changelog, so that I can make an informed decision about
   pulling it transitively into my build.
8. As a downstream/commercial consumer, I want confirmation that the package's own MIT
   license is unchanged and that the analyzer is build-time-only (not redistributed in my
   shipped output), so that I understand the practical scope of the LGPL dependency.
9. As the package maintainer, I want Sonar discovered by the existing analyzer-resolution
   step, so that its rules and default severities are extracted automatically rather than
   transcribed by hand.
10. As the package maintainer, I want the Sonar editorconfig generated by the existing
    merge generator, so that every rule is listed with its title, help link, and analyzer
    default, and stale rule IDs are detected on future version bumps.
11. As the package maintainer, I want all Sonar rules seeded to `none` and only a chosen
    subset promoted, so that the enabled set is a deliberate decision rather than a
    by-product of the analyzer's defaults.
12. As the package maintainer, I want an explicit, documented overlap policy ("existing
    analyzer wins; disable the Sonar twin"), so that promotion decisions are principled and
    repeatable.
13. As the package maintainer, I want a single foundational gap-analysis that classifies
    every Sonar rule as enabled / disabled-twin / out-of-scope, so that the per-rule work
    has an authoritative disposition to execute against.
14. As the package maintainer, I want every enabled Sonar rule covered by exactly one
    `[RuleDoc]` (a passing trigger test or, where genuinely untestable, a sourced
    `UntestableRules` entry), so that the existing coverage gate stays green and the
    rule-reference doc stays complete.
15. As the package maintainer, I want the rule-reference documentation to gain a
    "SonarAnalyzer.CSharp" section automatically when the editorconfig is regenerated, so
    that the published rule list reflects the new analyzer with no extra tooling.
16. As the package maintainer, I want the Sonar dependency version kept in sync between the
    package manifest and the central package-versions file, so that the existing
    dependency-sync check passes.
17. As the package maintainer, I want the analyzer-bump CI pipeline to remain idempotent
    after Sonar is added, so that its regenerate-and-commit-back step produces no spurious
    changes once the curated editorconfig is committed.
18. As the package maintainer, I want the repository's own dogfooding build to pass under the
    newly enabled Sonar rules, so that the solution continues to practise what it preaches.
19. As the package maintainer, I want to know early whether the analyzer resolver actually
    surfaces Sonar's DLLs, so that a packaging-layout mismatch surfaces as a checkpoint rather
    than as a silently empty editorconfig.
20. As the package maintainer, I want the README and AGENTS guidance updated from "four
    analyzers" to "five" with Sonar listed, so that the project's own documentation is
    accurate.
21. As the package maintainer, I want the changelog's Added section to record the new
    analyzer and its license, so that the (still unreleased) package's history is complete.
22. As a contributor executing the work, I want the effort decomposed into per-rule issues
    plus a handful of infra issues, so that progress is trackable in the established
    issues ledger.
23. As a contributor executing the work, I want the high-volume curation and test-writing
    produced via the existing batch-orchestration workflow, so that ~50–100 rules can be
    dispositioned and covered in one pass rather than hand-cranked.
24. As a contributor executing the work, I want the workflow to respect the harness's
    git-worktree incompatibility (parallel agents read-only; all build/test verification
    serial on the main checkout), so that the harness can find the repository root.
25. As a contributor executing the work, I want the adversarial-verification phase to enforce
    the "read the analyzer's report predicate before marking a rule untestable" bar, so that
    untestable claims are sourced and correct rather than convenient.
26. As a future maintainer hitting a Sonar version bump, I want newly introduced Sonar rules
    to arrive at `warning` (failing the coverage gate) rather than silently disabled, so that
    each genuinely new rule forces a deliberate keep-or-disable decision.
27. As a future maintainer, I want the overlap *policy* recorded durably in this PRD, while
    accepting there is no per-rule rationale store for the hundreds of disabled rules
    (consistent with the existing StyleCop treatment), so that the documentation burden stays
    proportionate.

## Implementation Decisions

**Scope.** Add `SonarAnalyzer.CSharp` only — pinned to the latest stable version at
implementation time (currently the `10.27.x` line). VB.NET (`SonarAnalyzer.VisualBasic`) and
any separate styling package are out of scope: the repository is C#-only end to end
(resolution requests C# analyzers, every test is C#).

**Licensing.** `SonarAnalyzer.CSharp` is LGPL-3.0-only. Decision: **accept and document.**
The package's own license remains MIT (declaring a dependency does not relicense it); the
analyzer runs at build time and is not linked into a consumer's shipped binaries, so the
copyleft obligations are minimal in practice. The README's bundled-analyzers listing gains a
Sonar row with a license note, and the changelog records it.

**Dependency wiring.** Sonar is added to the package manifest's dependency list **and** the
central package-versions file, version-matched, mirroring the existing analyzer entries. The
existing dependency-sync check (and its CI step) enforces the match.

**Analyzer-resolution allow-map (the only code change).** A single entry is added to the
resolver's package allow-map, mapping the Sonar package id to its editorconfig file name.
No other production code changes. The resolver's existing DLL-selection logic already handles
Sonar's packaging layout (`analyzers/dotnet/cs/…`, including roslyn-versioned variants), and
the descriptor extractor already handles multi-DLL packages with id de-duplication — both are
exercised, not modified.

**Editorconfig seed.** A new per-analyzer editorconfig is generated by the existing merge
generator (listing every Sonar rule with title, help link, and analyzer default), then a
**one-time normalization** flips every rule to `none`. This mirrors the StyleCop precedent
and the prior one-time normalization effort, and requires **no generator change**. Because
the generator preserves the curated severity line on subsequent runs, future version bumps
surface only *new* rule ids at `warning`.

**Enabled set — broader maintainability set (~50–100 rules).** Promotion is anchored on the
complexity metrics (cognitive complexity, cyclomatic complexity, nesting depth, method
length, parameter count — none of which the other four analyzers provide) and extends to a
curated body of Sonar maintainability and design rules.

**Overlap policy — existing analyzer wins.** Where a Sonar rule is an exact twin of a rule
already enforced by NetAnalyzers/StyleCop/Meziantou/IDE, the Sonar rule stays `none`. Sonar
is promoted only where it is additive or materially distinct. Whole-program
symbolic-execution rules are de-prioritised: most overlap the already-enabled nullable
reference-type checks or CA rules (so are disabled under this policy), and they carry the
highest build-time cost.

**Gap analysis (foundational).** A single authoritative disposition classifies all Sonar
rules into enabled / disabled-twin / out-of-scope. The hundreds of disabled rules are
dispositioned in bulk here; they do **not** get individual issues. This disposition blocks
the per-rule promotion work.

**Coverage model.** Every promoted rule gets exactly one method-level `[RuleDoc]` plus a
trigger test, or — where a rule genuinely cannot fire in the single-project build harness — a
class-level `[RuleDoc]` in the untestable set with a sourced root-cause reason. The existing
coverage gate enforces one-doc-per-active-rule and rejects docs pointing at inactive rules.

**Documentation regeneration.** The rule-reference document gains a "SonarAnalyzer.CSharp"
section automatically when regenerated (the generator discovers any per-analyzer editorconfig
and strips the `Analyzer.` prefix for the section title). README and AGENTS guidance move from
"four analyzers" to "five"; the changelog Added section records the analyzer and its license.

**Delivery & decomposition.** Full PRD broken into **per-rule issues** (~50–100, one per
*enabled* rule) plus ~4 infra issues (dependency + sync, allow-map entry, seed-and-normalize
editorconfig, docs/changelog/rule-reference) plus the one foundational gap-analysis issue. All
issues are worked on a single feature branch and delivered as **one pull request**; issues
serve as the workflow's work-list and tracking ledger (moved to the done folder as completed).

**Production method.** The curation and test-writing are produced via the repository's
two-phase batch-orchestration workflow: parallel **read-only** investigators draft
dispositions and trigger sources, an adversarial skeptic verifies twin calls and untestable
claims, then a **serial** phase applies edits, builds, runs the harness, and commits on the
main checkout (no git worktrees — the harness locates the repo root by a `.git` *directory*
and breaks under worktrees). Launching the workflow is gated on explicit opt-in at execution
time.

**No new module.** The deep-module check was performed and concluded that no new code module
is warranted: the analyzer-ingestion pipeline (resolve → extract → merge → reference) is
already a set of deep modules, and Sonar flows through them unchanged. Adding Sonar validates
the existing design rather than extending it.

## Testing Decisions

**What makes a good test here.** Tests exercise *external behaviour*, not analyzer internals:
each one packs the real package, builds a throwaway project that references it, and asserts on
the SARIF build output (e.g. that a specific rule id is reported on a crafted violation, or —
for opt-out/negative cases — that it is not). Tests are driven entirely through the public
build pipeline, never by invoking analyzers directly.

**Modules tested.** Every *enabled* Sonar rule is tested — a crafted source snippet that
triggers the rule, asserting the rule id appears in the build output. Negative/opt-out probes
(where relevant) assert absence and carry no `[RuleDoc]`. Disabled and out-of-scope rules are
not tested (they are not active). The coverage gate is itself the meta-test that the enabled
set and the `[RuleDoc]` set agree.

**Test organisation.** Sonar tests live in a new subfolder mirroring the existing
third-party-analyzer layout used for Meziantou — a folder named after the originator, with
group files named `<OriginalClass><Group>Should.cs`, namespaces mirroring the folder, each
file kept under 1000 lines and split when it grows past that. Tests use the shared
package-fixture collection and the common test base, as the existing rule tests do.

**Prior art to model on.**
- Third-party analyzer with many rules organised into a subfolder: the Meziantou analyzer test
  classes (core/extended split).
- Per-rule positive trigger tests with method-level `[RuleDoc]` and SARIF assertions: the
  CodeAnalysisRules test classes (e.g. the security group) and the StyleCop analyzer tests.
- Coverage/reconciliation gate: the rule-doc coverage test.
- Untestable rules with sourced root-cause reasons: the untestable-rules class.
- Build/throwaway-project harness and helpers: the project-builder, package-fixture, and
  build-output helpers.

**Untestable bar.** A promoted rule may only be recorded as untestable with a sourced reason
explaining why it cannot fire in the single-project harness (e.g. it requires multi-project or
specific-framework context). The confounder playbook applies first — real package references
and stub types often unblock rules that look untestable — so untestable should be the
exception, not the default, for Sonar (whose rules generally emit their own ids at build time).

## Out of Scope

- VB.NET analysis (`SonarAnalyzer.VisualBasic`) and any separate Sonar styling package.
- Replacing or retiring any of the existing four analyzers; their curated baseline is left
  untouched except where an exact Sonar twin would otherwise double-report.
- SonarQube / SonarCloud server integration, the Sonar scanner, or any connected-mode
  features — only the standalone, offline NuGet analyzer is in scope.
- Changes to the editorconfig merge generator, the resolver's DLL-selection logic, or the
  rule-reference generator — all are reused as-is (one data entry aside).
- A bespoke overlap-detection tool/script — overlap is judged within the batch workflow, not
  by new tooling.
- A per-rule rationale store for disabled rules (consistent with the existing StyleCop
  treatment); only the policy is documented, in this PRD.
- A version bump / release cut — the package is still pre-release; Sonar is added under the
  existing Unreleased changelog section.
- Promoting whole-program symbolic-execution bug rules as a category (de-prioritised on
  overlap and build-cost grounds).

## Further Notes

- **Dogfooding can break the repository's own build, not just test coverage.** Promoting a
  rule at enforced severity means the solution's own source must satisfy it. Some promotions
  will require fixing repository code, and a few may justify reconsidering the rule if the
  repo's own code cannot reasonably comply. Expect a long red period in the single PR until
  *both* code and coverage catch up.
- **Resolver-discovery checkpoint.** The first infra issue must confirm the resolver actually
  surfaces Sonar's analyzer DLLs and the extractor returns the full descriptor set; a layout
  mismatch would otherwise yield a silently empty editorconfig.
- **Build-time cost.** Sonar adds analysis cost; the gap analysis should prefer syntactic /
  maintainability rules and treat expensive whole-program rules with caution. Analyzer
  execution reporting is already enabled, so cost is observable.
- **Analyzer-bump idempotence.** The initial PR changes both the manifest and the
  package-versions file, which triggers the analyzer-bump pipeline. Because the curated
  editorconfig (with its `none` + promoted severities) is committed as part of the PR, the
  pipeline's regenerate step preserves those severities and produces no spurious diff.
- **Execution gating.** This PRD authorises the plan; the batch-orchestration workflow is only
  launched after explicit opt-in. No analyzer files, dependencies, or editorconfigs are
  touched until the issues are created and execution is approved.
